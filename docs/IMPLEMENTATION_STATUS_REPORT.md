# CRM Implementation Status Report

> Generated by audit on 2026-06-19. Read-only audit — no application code, schema, or mock data was modified, created, or deleted. The only file produced by this exercise is this report.

---

## 1. Executive Summary

**Overall project status:** The CRM is a substantial, architecturally ambitious build. Read paths are broadly real and DB-backed across almost every module (Finance, Settings, Approvals, Masters). The system's biggest weakness is not "missing pages" — it's **missing write APIs** in Finance, **fragmented RBAC enforcement**, and a meaningful amount of **dead/orphaned code** from superseded design iterations that was never cleaned up.

**What is production-ready:**
- Core CRM: Pipeline (leads/opportunities/tasks), Daily Updates, KRA tracking, Employees, Reviews — live UI + live API + DB, with write actions.
- Settings suite (Organization, Identity, Finance Admin, Master Data config, Communication Center, Integration Center, Security Center, Policy Engine, Performance, CRM Admin) — all real, DB-backed, server-permission-checked at the page level.
- Global Approval Inbox (`/approvals`) and the underlying `workflow-engine` — fully real lifecycle (start/approve/reject/return/delegate/cancel), used by Pipeline (large-deal/discount approval), Finance Advances, and Expenses.
- Finance read surfaces (Dashboard, Bank Book, Cash Book, Expense Register, Vouchers, Conveyance list, Advances list) — live API + live UI, read-only.

**What is partially ready:**
- Finance write workflows: only one real POST exists in the entire `/api/finance/*` surface (`POST /api/finance/advances`). Everything else — expense create/edit/approve, voucher creation/cancel/PDF, bank/cash entry, transfers, conveyance trip logging/settlement — is UI-gated with "coming soon" messaging, not backed by an API.
- RBAC: three coexisting permission systems (`roles.ts` legacy predicates — enforced; `rbac.ts`/`AppRole` matrix — built but not enforced anywhere; `access-control`/`Role`-`Permission` — enforced on Admin/Identity/Masters/CRM-admin routes only). Confirmed real gaps: customer-master `PATCH` has no permission check while its sibling `DELETE` does.
- Mobile app: most navigation/data-sync screens are real; Expense Claim and Local Conveyance (including all GPS/Maps capability) are explicitly mock/placeholder.

**What is mock/demo only:**
- Finance: Reports page, Tally Export, Customer Profitability, Budget, Recurring Expenses, bank statement import/reconciliation UI, conveyance monthly settlement/policy tabs.
- Mobile: Expense Claim screen, Local Conveyance screen (mock GPS), no Approvals screen on mobile at all.
- A second, fully-built but **unreachable** "Admin Console v1" component set (`settings/data/adminModules.ts` + 3 components) containing fabricated fake "recent activity" entries (named individuals) that could mislead anyone reading the code cold.
- An orphaned mock approval-engine module (`ApprovalEngineClient.tsx` + ~6 sibling components) superseded by the real `WorkflowCenter`, but not deleted.

**Biggest risks (ranked):**
1. **RBAC fragmentation** — an admin editing permissions in the legacy `RolesClient` screen and the new `Identity` screen can get two different, unsynced answers for the same employee's access. Combined with at least two confirmed unguarded write/read routes, this is the top pre-production risk.
2. **No soft delete anywhere** — all 108 Prisma models use hard deletes. Any UI delete action is unrecoverable; this compounds the RBAC gap (an under-permissioned route could destroy data with no audit trail to undo it).
3. **Money fields are Float, not Decimal**, across ~40+ finance fields — rounding/precision risk once real write volume starts, especially before any reconciliation feature exists to catch drift.
4. **Finance is read-only in practice.** Every page that "looks" functional (Dashboard buttons, Voucher actions, Expense Register actions, Conveyance trip logging) is cosmetically complete but functionally a dead end — a real risk of being mistaken for "done" by anyone not reading the gating logic.
5. **Mobile GPS/Maps and Expense/Conveyance capture are 0% implemented** despite a fully-specified design doc (`GOOGLE_MAPS_INTEGRATION.md`) — the schema is ready (`TravelClaim` lat/lng/distance columns exist) but no code consumes it.

**Recommended next step:** Do not expose Finance write actions or mobile Expense/Conveyance capture to end users yet. Prioritize: (1) resolving the RBAC duplication (pick one system, retire the other, audit every route against it), (2) wiring the four missing high-value Finance write APIs (Expense CRUD, Voucher creation, Bank/Cash entry, Advance disburse/settle), (3) deleting the confirmed-dead code so the codebase doesn't mislead future Claude Code sessions or new developers.

## 5. Pending Development Items

### Critical
- **Resolve RBAC fragmentation.** `rbac.ts`/`AppRole` matrix is built and DB-seeded but not enforced anywhere; `access-control` enforces only Admin/Identity/Masters/CRM-admin; legacy `roles.ts` predicates gate everything else. Pick one as the system of record, retire the others, audit every route. *Files: `src/lib/roles.ts`, `src/lib/rbac.ts`, `src/lib/access-control/*`, all `src/app/api/**/route.ts`.* Phase: before any further write-API work.
- **Fix unguarded routes.** `customers/master/[id]` PATCH and `admin/masters` GET/POST have no permission check. *Files: `src/app/api/customers/master/[id]/route.ts`, `src/app/api/admin/masters/route.ts`.* Phase: immediate, pre-production.
- **No soft delete anywhere.** All 108 models hard-delete. Combined with the above gaps, any RBAC miss is unrecoverable data loss. *Files: `prisma/schema.prisma` (all models), every DELETE handler.* Phase: before opening write access broadly.

### Important
- **Finance write APIs.** Expense CRUD, Voucher creation/cancel/PDF, Bank/Cash entry + transfer, Advance disburse/settle, Conveyance trip log/submit/settlement. *Files: `src/app/api/finance/**`, `src/lib/finance-engine/voucher.ts`.* Phase: Step 3-4 per existing wiring plan.
- **Reconcile two voucher-numbering systems** (`VoucherSequence` vs `VoucherConfiguration.generateVoucherNumber()`) before any voucher-creation endpoint ships — currently neither is wired to a POST, so the conflict hasn't surfaced yet but will the moment one is built.
- **Wire approval notifications.** `approveRequest`/`rejectRequest`/escalation never call the (fully built) communication-engine — approvers/requesters get no notification, only a silent audit-log row. *Files: `src/lib/workflow-engine/approval.ts`, `src/lib/communication-engine/*`.*
- **Wire or remove escalation.** `checkAndTriggerEscalations()` is fully implemented but never invoked by any cron/route — escalation rules configured in Settings currently have zero runtime effect. *File: `src/lib/workflow-engine/escalation.ts`.*
- **Decimal migration for money fields.** ~40+ Float fields across Voucher/Expense/Collection/Payment/EmployeeAdvance/TravelClaim. *File: `prisma/schema.prisma`.* Phase: before meaningful write volume / before Hostinger MySQL cutover.
- **Customer Master duplication.** `/customers` (legacy) vs `/masters/customers` (global) — same `Customer` table, two UIs. *Files: `src/app/customers/**`, `src/app/masters/customers/**`.*
- **Mobile GPS/Maps + Expense/Conveyance capture** — 0% implemented despite a complete design doc and ready schema columns. *Files: `src/app/mobile/screens/ExpenseClaimScreen.tsx`, `ConveyanceScreen.tsx`; doc: `docs/modules/finance/GOOGLE_MAPS_INTEGRATION.md`.*
- **Mobile Approvals screen** — specified, never built. *Doc: `docs/modules/finance/MOBILE_REQUIREMENTS.md`.*

### Later
- Tally Export, Customer Profitability (real query), Budget, Recurring Expenses — all zero-code today; scope and build once core Finance writes are stable.
- Bank statement import / reconciliation — UI shells exist over mock data; needs a real parser + matching engine.
- Number Series Master, Tax Master, Cost Center Master — schema/UI work, needed before Vouchers can carry real cost-center/tax data (voucher detail route already hardcodes `costCenter: null`).
- Security Center / Integration Center enforcement — currently config-only, by design; wire into runtime auth/integration calls when prioritized.
- SLA timeout consistency — Approval Inbox hardcodes 48h client-side instead of reading `WorkflowStep.timeoutHours`.

## 6. Items Needing Alteration / Refactor

- **Duplicate routes:** `/customers` vs `/masters/customers` (same table, two UIs).
- **Misplaced/duplicate masters layer:** `/settings/masters` (governance) coexists with `/masters/*` (records) — functionally distinct but the "Masters" label appears in three nav locations, confusing for new users/devs.
- **Old admin panel:** `/admin` → `/settings/administration` (`AdminClient.tsx`) is fully functional but navigationally orphaned (not in `AdminConsole`'s card list); still duplicates Roles & Approvals concepts rebuilt elsewhere. Either redirect it away or remove per the architecture plan's own "Phase 6 complete" trigger.
- **RBAC conflict:** legacy `RolesClient`/`AppRole` 4-boolean matrix vs new `Identity`/Module×Action×DataScope model — both live, unsynced. Real production risk: an admin could revoke access in one screen while the other still grants it.
- **Approval engine placement:** the *real* approval-engine UI (`WorkflowCenter`) lives at `/settings/workflow/approval-engine`, but a parallel, mock, unreachable `ApprovalEngineClient.tsx` plus ~6 sibling components sit in the same folder — confusing for anyone navigating the code. `data.ts`/`ApprovalInbox`/`ApprovalDetailDrawer` from that same folder are genuinely shared and alive (used by `/approvals` and `/finance/approvals`) — relocate those three to a shared location before deleting the rest.
- **Finance Approvals reuses global components correctly** but duplicates ~80 lines of fetch/mapping logic — extract a shared `useApprovalInbox()` hook.
- **Dead "Admin Console v1" prototype** (`settings/data/adminModules.ts` + `AdminModuleCard.tsx`, `AdminStatsCard.tsx`, `RecentChanges.tsx`) — unreachable, contains fabricated fake activity data with realistic-looking employee names. Should be deleted, not left as ambiguous dead code.
- **Two orphaned Finance schema models:** `FinancePolicy` and `VoucherConfiguration` — confirmed zero references outside their own admin-engine CRUD; either wire them in or document them as reserved-for-future.
- **Finance mock pages/data files still referenced:** `bank-book/data.ts`, `cash-book/data.ts`, `expenses/data.ts`, `expenses/categories/data.ts`, `conveyance/data.ts`, `finance/_shared/transferStore.ts` (in-memory, resets on reload) — all should be retired as their corresponding write APIs ship.

## 10. Final Verdict

- **Can this be used by users now?** Yes, for read-heavy and core-CRM workflows (Pipeline, Daily Updates, KRAs, Employees, Settings administration, Approval Inbox). No, for any Finance write action or mobile expense/conveyance capture — those are UI-complete but functionally inert.
- **Which modules can be safely shown?** Core CRM (Pipeline/Leads/Opportunities/Tasks/Daily Updates/KRAs/Employees), all of Settings (with the caveat that Security/Integration Centers are config-only, not enforcing), Global Approval Inbox, Finance Approvals, and all Finance *read* surfaces (Dashboard, Bank Book, Cash Book, Expense Register, Vouchers, Advances list, Conveyance list) as reporting/visibility tools.
- **Which modules must remain disabled?** Finance Reports, Tally Export, Customer Profitability, Budget, Recurring Expenses, bank statement import/reconciliation, conveyance monthly settlement/policy tabs, mobile Expense Claim, mobile Local Conveyance, and any Finance "create/edit/approve/disburse/settle/cancel" button — all of these are already gated client-side, which is correct; they should stay gated until their APIs exist.
- **What must be completed before Finance production use?** (1) RBAC consolidation and the two confirmed unguarded-route fixes, (2) soft-delete on Finance-critical models, (3) the Decimal money-field migration, (4) the missing write APIs listed in §5/§8 (Expense CRUD, Voucher create/cancel, Bank/Cash entry, Advance disburse/settle, Conveyance trip workflow), and (5) resolving the dual voucher-numbering mechanism before any voucher-create endpoint ships.
